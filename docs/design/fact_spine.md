## 0. 一句话

**在起草之前，从源建一张完整、归并到单一真相、冻结的实体登记表（Fact Spine），让 60 章并发起草受其硬约束消费、不得自由现编任何实体事实**；事后一致性检测降级为薄安全网。一致性问题从「21 万字 prose 空间」搬到「~50-200 行 registry 空间」，并前移到起草前——这是它可解、而 prose 空间对账不可解（A2 18% 召回）的根本原因。

## 1. 问题与定位

### 1.1 承重崩的根因（已被 5 本 Opus 精读锁死）

线性甜区 5 本承重全线 31-58，**每一类失败都是同一个根**：同一实体在不同章被写成不同事实。

| 失败类 | 5 本实证 |
|---|---|
| 同人多名 | 生父=徐正鸿/正清/山川/景德（怀孕）；顾明骁↔顾明景（退婚） |
| 同名多身份 | 顾明乾=养父反派/亲二哥/双胞胎弟（退婚）；宋时奕=武王族弟/九皇子/太子（傲世） |
| 身份漂移 | 男主张冠李戴：冷面知青孙成才 ch6 出局、真男主另一人（六零） |
| 数值矛盾 | 走失 20/26/17/23 年（退婚）；婚龄五/十/三年（六零）；军衔六称谓（误嫁） |
| 事件双版本 | 绑架 ch24/25 互斥（误嫁）；生子后又怀孕（误嫁） |
| 设定漂移 | 母亲身份三易名（傲世）；空间金手指三变形（六零）；1975 饥荒 vs 1970 东北（六零） |

根因链（记忆 R13 早已诊断、本轮再证）：**并发分章起草 + plan brief 抽象 → 起草忠于抽象 brief、自由现编具体事实 → 60 个并发 draft 各编各的**。canon/bible 只钉了主角+少数主角团，配角长尾（顾明乾/徐父/宋时奕）从不在表内，且即便在表内也是 plan 级 advisory、起草在 prose 里照漂。

### 1.2 为什么「事后对账」只能是创可贴

1. **能检不能裁**：60 章各写各的之后，对账能查出「徐父 4 名」但无权威决定哪个真——真相当初没钉在任何地方。
2. **修即引漂**：项目最硬旧教训（架构 A）——LLM 重写引入的漂移 > 它修复的，打地鼠。
3. **太晚太贵**：¥21/本已花在注定带病的 draft 上，只能止损。
4. **检测本身也漏**：A2 朴素 1M 对账召回仅 18%（深处不查）；语义级身份漂移需 LLM 判=不可靠。

### 1.3 为什么「事前做一半」（现有 canon/身份账）没堵住

架构 A 钉了 canon、R14 加了 `ledger.py` 身份/生死/修为账——但今天照崩。三缺口：

- **覆盖缺口**：登记表从不完整（只主角团，配角长尾缺席）。
- **绑定缺口**：账本是 plan 级、作为「前情账本 context」注入 = advisory，不是硬冻结（`plan 审计干净 ≠ prose 正确`）。
- **真相源缺口**：canon 由会脑补的 mining 建；徐父名字若没在 canon 里**归并成单一值**，起草看到的就是「无权威」→ 自由发挥。

**事前-完整 = 覆盖全实体 + 硬冻结绑定 + 建表时归并成单一真相。三缺口同时补，才是真事前。**

## 2. 核心原理（设计的命门）

> **把一致性问题从 prose 空间搬到 registry 空间，并前移到起草之前。**

- **prose 空间**（21 万字、语义、跨段长程）：矛盾不可枚举、检测召回低（A2 18%）、修复引漂。不可解。
- **registry 空间**（~50-200 行结构化实体表）：矛盾**可枚举**（每名一身份、每数值一值或一单调规则、每事件一次）、可确定性裁决 + 定向 LLM-verify、改一张表不碰正文。**可解。**

并发起草天然跨章漂移（无人看见别人写了啥）。并发下拿一致性只有两条路，顺序起草已被 M0 证伪（章缝 -21% 远不达减半线 + 16× 墙钟），故唯一可行路 = **共享冻结 canon + 受约束并发起草**。Fact Spine 即此路的完整执行。

## 3. 架构：五阶段

```
源 clean.txt
  │
  ├─[阶段1] 全量实体抽取（复用 mining map-reduce，扩 schema）────────┐
  │                                                                  ▼
  │                                                       raw_registry（含矛盾/多值）
  │                                                                  │
  ├─[阶段2] 登记表级归并裁决（registry 空间，确定性+LLM-verify，一次）│
  │   同人多名聚类→规范名 · 每名钉一身份 · 每数值一值/单调规则 ·      │
  │   事件唯一时间线 · 生死/孕育序                                    ▼
  │                                                       FACT SPINE（冻结·单一真相·版本化资产）
  │                                                                  │
  ├─[阶段3] plan/macro 从 Spine 生成（节拍引用规范实体+单一时间线）   │
  │                                                                  │
  ├─[阶段4] 起草受 Spine 硬约束（扩 ledger.state_before/format_context│
  │   注入本章实体切片=冻结事实，禁新增/改名/矛盾）──→ 60 章并发 draft │
  │                                                                  ▼
  └─[阶段5] 事后薄网（复用 prose_facts.cross_check 进交付门，兜残漏）→ 成品
```

### 3.1 数据契约：Fact Spine schema（落 `schemas.py`）

```python
@dataclass
class Entity:
    canonical_name: str            # 唯一规范名（冻结，起草只能用这个）
    aliases: list[str]             # 已归并的别名/旧名（起草禁用，归一目标）
    kind: str                      # 人物 / 势力 / 设定(金手指·体系) / 地点
    identity: str                  # 单一身份/立场（顾明乾=亲二哥，钉死一个）
    gender: str                    # 配角性别钉死（治性别翻转）
    faction: str
    relationships: list[dict]      # [{to, type}]，type∈{父/兄/弟/夫妻/师徒/敌/...}
    attributes: dict               # {age, rank, ...} 单值；或 {rank: {rule:"单调↑", seq:[...]}}
    status: dict                   # {alive: bool, dies_at: ch|None}（生死+时点，治复活）
    arc: str
    first_ch: int                  # 首次登场章（治重复初遇）

@dataclass
class FactSpine:
    entities: list[Entity]
    timeline: list[dict]           # [{event_id, ch, when, unique:bool}] 单一有序事件骨架
    settings: list[dict]           # 金手指机制/战力体系层级（单一定义，治三变形）
    reconcile_log: list[str]       # 归并裁决留痕（哪些名被并、哪些冲突怎么裁）— A6 版本化
```

落盘 `output/<源>_full/fact_spine.json`，与 bible/macro/plan 并列为一等规划资产。

### 3.2 阶段 1：全量实体抽取（复用 + 扩展 `mining.py`）

- 复用现有 map-reduce（EXTRACT_CHUNK 12 窗并发 → REDUCE）。**扩 EXTRACT_CHUNK 提示词**：每窗除主角/爽点外，抽**全部命名实体**→{名、出现别称、身份、关系、提到的数值(年龄/军衔/年限)、生死线索}。flash 走量、局部抽取=模型强项（A2' 已证）。
- REDUCE 阶段汇总成 `raw_registry`（此时**允许含多值/矛盾**——同人多名、同名多身份都原样留着，留给阶段 2 裁）。

### 3.3 阶段 2：登记表级归并裁决（**本架构的智力核心**）

在 `raw_registry`（结构化、小）上做一次性归并，输出冻结 Spine。每类矛盾有确定性主裁 + 定向 LLM-verify 兜：

| 矛盾类 | 裁决法（registry 空间，复用现有件） |
|---|---|
| **同人多名** | 复用 `prose_continuity.PROSE_CLUSTER + PROSE_NAME_VERIFY`（带例句核查、句级共现、称谓后缀停用表）聚类→选规范名（高频/显式）→其余进 aliases。**已验证防过并**（高志强/高志坚兄弟不并）。 |
| **同名多身份** | 一名只能一身份。冲突→`IDENTITY_VERIFY`（带源引文）判：(a) 抽取错→选真身份；(b) 真有两人共名→拆成两实体、规范名加消歧后缀（顾明乾→顾明乾/顾明乾·弟，或更名）。冻结决定。 |
| **数值矛盾** | 单值属性（年龄/婚龄/走失年数）→选规范值；渐进属性（军衔/修为/时间流逝）→记 `{rule:单调↑, seq}`，复用 `audit.fix_power_monotonic` 的单调逻辑。 |
| **事件唯一/序** | 复用 `ledger.validate_timeline + dedup_first_meetings`，但**输入从 plan 场景元数据上移为 Spine 时间线**：每 event_id 一次、生死→孕育→出生序、初遇唯一。 |
| **设定漂移** | 金手指/体系→单一定义（购物群 vs 团购 App 选一套机制冻结）。 |

> **杠杆所在**：以上全部在 ~50-200 行表上操作，矛盾可枚举、可确定性消解、改表不碰正文。这正是 A2 在 21 万字 prose 上做不到（18% 召回）、而这里能做到的根本差别。归并留痕进 `reconcile_log`（A6）。

### 3.4 阶段 3：plan/macro 从 Spine 生成

`_plan_macro / _plan_one_chapter` 的 bible 输入换成**冻结 Spine 切片**：节拍只能引用规范实体名、单一时间线。从源头排掉 plan 级排程矛盾（绑架演两章、生子后又怀孕、预告跳空）——这些今天靠铁律 prompt 压不住（记忆实证），从 Spine 生成则结构上不产生。

### 3.5 阶段 4：起草硬约束注入（扩展 `ledger.py`）

`ledger.format_context` 今天已注入「已死亡绝不复活/当前修为绝不写低」=硬约束模板（已验证有效）。**扩展为 Spine 切片**：每章起草前，从 Spine 取本章登场实体 →

```
【本章冻结事实·不得改写/新增/矛盾】
- 角色规范名(只能用这些,禁用别名): 顾明乾、安宁、陆擎泽...
- 身份(钉死): 顾明乾=安宁亲二哥(非养父非弟) | ...
- 关系: 安宁↔陆擎泽=夫妻 | ...
- 数值(钉死): 安宁走失=17年 | 顾巍军衔=团长 | ...
- 生死: 张三=已于第N章死亡,本章绝不在场 | ...
- 设定: 空间=购物群App(机制见下),禁改写为其它形态
brief 需要而上表没有的事实,一律不得现编;按"服从冻结事实 > brief 叙述"。
```

DRAFT 铁律加：**冻结事实优先级 > brief 叙述**（R13 主刀「plan brief 事实清单化」的彻底版——喂的是冻结 Spine 切片而非再生 brief）。

### 3.6 阶段 5：事后薄网（复用 `prose_facts.cross_check` 进交付门）

起草仍可能不听话（LLM 非完全服从）。保留 `prose_facts.fact_table_audit`（逐章抽→代码跨章对比，已实现）作**薄安全网**：**现在它对照的是冻结 Spine**（不再是无锚自检），命中=「起草违反了冻结事实」→ 高置信、进 `ship_issues`（生死类已是高置信，身份/数值随 Spine 锚定后误报大降）。**事前为主、事后兜底——互补不二选一。**

## 4. 与现有件的关系

| 现有件 | 关系 | 改动 |
|---|---|---|
| `mining.py` EXTRACT_CHUNK/REDUCE | **扩展** | 抽全量实体进 raw_registry |
| `prose_continuity.py` 聚类+verify | **复用** | 同人多名裁决器（已防过并） |
| `ledger.py` validate_timeline/dedup/state_before/format_context | **复用+上移** | 输入从 plan 元数据上移为 Spine；format_context 扩为 Spine 切片硬约束 |
| `audit.fix_power_monotonic` | **复用** | 渐进属性单调规则 |
| `prose_facts.py` cross_check | **复用+翻转** | 从事后 advisory 翻转为「对照冻结 Spine」的交付门薄网 |
| `schemas.py` | **新增** | Entity / FactSpine 契约 |
| `produce.py` | **接线** | 阶段 2/3/4 插入、阶段 5 进门 |

> 关键认知：**检测半边（prose_facts cross_check）和注入半边（ledger format_context）都已存在且验证过**。Fact Spine 不是从零造，而是**补上中间缺的「完整抽取 + 归并裁决 + 冻结」，并把检测从事后翻转到事前**。

## 5. 八公理对齐

- **A2 场景原子**：保留并发分章（不退顺序起草，已证伪）；用共享冻结 Spine 解并发漂移。
- **A4 源是脊柱**：Spine 即「脊柱」的字面实现——事实从源抽、冻结、起草消费而非现编。
- **A6 版本化资产**：fact_spine.json + reconcile_log 落盘、版本化；归并裁决可复现、可审计。
- **A7 成本自适应**：建 Spine 是一次性、对硬源可加深抽取/裁决（多 pro 调用），但远低于产 60 章被拒的浪费。
- **A1 文件即状态**：Spine 是阶段间冻结契约，阶段即纯函数消费它。

## 6. 诚实边界（不治什么）

1. **不抬笔力/人**：Spine 治承重（事实一致性），把它从 31-58 抬向 70+。笔力/人是生成侧天花板，只有微调（spec §18-K）。总分仍被 ~75 卡——但今天瓶颈正是承重，这刀打在正地方。
2. **抽取仍由 LLM**：Spine 由 mining 建，会有抽取错/漏实体。但全源 map-reduce 抽结构化表是模型强项（A2' 18%→结构化更强已证），且表小可确定性核验（每名一身份、每数值一值），远比要求起草维持 60 章一致可靠。漏抽的实体→阶段 5 薄网兜。
3. **源级真歧义**：源本身就矛盾（顾明乾双胞胎 POV 漂移）→ 归并必须裁一个规范真相，裁错则「一致但可能轻微偏源」（仍优于不一致）。裁决质量取决于阶段 2 的 verify。
4. **不保证零残漏**：起草可能违反约束→薄网降级残漏为 advisory/拦截，非清零。

## 7. 分阶段落地 + 可证伪判据

> 纪律：不一次性切换；每阶段先在已知病例上测，过判据才接线（沿用 tier3 M0 方法论）。

**M0 — 归并裁决可行性（先验证智力核心，最便宜）** ✅ 已跑（2026-06-13，¥2.07，`scripts/m0_spine_reconcile.py` / `output/m0_spine_reconcile.json`）
- 在 5 本**成品正文**（最难的事后漂移数据）上跑现有复用件（`cluster_names` + `prose_facts.cross_check`）。
- **结果**：头号病例消解 ~30-40%（< 70% 判据），但**精度高、零错并**（verify-gate 守住）。缺陷类干净二分：
  - ✅ 现有件治得了（事后即可，可当 §3.6 薄网）：**同名多身份**（同一名跨章冲突角色：宋时奕族弟/九皇子/太子、君怀柔、顾明乾部分）、**数值有 key**（婚龄五/十/三、年龄、彩礼）、**生死复活**（徐景好 ch14死→ch15活、许琉吟——Opus 都漏的，det 抓到）、**edit-1/粘连**（魏巍→顾巍）。
  - ❌ 现有件结构上治不了（事后无解）：**同一角色→多个不相似名**（生父徐正鸿/正清/山川/景德、外公沈/庄/宋、母亲三易名、顾明骁↔景）——`cluster_names` 名差 2-3 字并不动、verify 正确判「漂移正文里就是不同人」；**事件双版本**（绑架 ch24/25）；**角色级数值未被抽**（走失年数）。
- **关键裁定**：M0 在事后数据上测，头号杀手类（同角色多名）正是事后无解类——而它恰是**事前冻结脊柱构造上不产生**的类（源里生父=一实体、钉死规范名、60 章全用它、四名漂移根本不发生）。**故 M0 的 <70% 是对「事后对账」的判决（本就是创可贴），反而证实事前脊柱的必要性。** 同时验证：归并精度安全（零错并）、cross_check 三类可复用进薄网。
- **对 spec 的两处修正**：①§3.3 阶段2 需**新增「角色锚定规范名分配」裁决器**——在 SOURCE 建表时，每个唯一角色（生父/外公/男主）→ 钉死一个规范名（源里实体出现一次，无歧义），这是 `cluster_names` 名相似度归并覆盖不到的、且只能在事前 registry 空间做；②§3.6 薄网的 `cross_check` identity **误报高**（角色 vs 关系：医生/顾夫人、嫡女/堂姐都报），进交付门前需加「是否真矛盾」过滤（同义/角色+关系不算）；③ FACT_EXTRACT 的 number key 扩到角色级事实（走失/失散年数）。

**M1 — 端到端（Spine 驱动起草）单本对照** ◐ 已跑一轮（2026-06-13，退婚财阀，`HIKI_SPINE=1`，`scripts/m1_run.py`+`m1_compare.py`）
- 实现：REDUCE_BIBLE 加完整配角表+`key_relation`+唯一角色锚定；`produce._spine_map/_spine_block` 把全量角色规范名编成「角色名钉死」硬约束（禁改名/新造名/改身份血缘）注入 `_control_plane`，behind `HIKI_SPINE=1`；规划 `bible_brief` 也喂全量冻结角色表。
- **结果（cross_check A/B）= inconclusive，未达「承重矛盾减半」判据**：raw 漂移 基线46 vs Spine47（无减少）。但拆开看：
  - **cross_check「身份」~90% 是误报**（角色vs关系/同义/剧情反转），raw count 噪声淹没信号 = **错的测量仪**（M0 已警告，M1 量化坐实）。
  - **拆出真矛盾后名字/身份子类有改善**：基线「可乐 女儿ch27 vs 儿子ch39」性别翻转 → Spine 版消除；顾家三人身份纠缠 → Spine 版相对干净；异名归一只剩零星（陆擎川→陆擎泽），名钉住了。
  - **数值子类未改善甚至更多（6→8）**：本版 Spine 只钉角色名+身份，**没钉数值属性**（彩礼30/60/15万、年龄22/21/24 仍漂）。
- **诊断**：未达判据的两因——(a) 测量仪 cross_check 身份误报淹没信号，需「真矛盾过滤」或 **Opus 承重维精读 A/B**（本应是主测量#2，本轮未做）；(b) 本版 Spine 覆盖不全，只实现「名钉死」，未实现「数值/时间线登记表」（阶段2 部分缺）。
- **M1.5 下一步（干净裁定所需）**：① cross_check 加真矛盾过滤（排角色+关系/同义）② 扩 Spine 到数值/时间线事实（彩礼/年龄/婚龄/走失年数钉单值或单调规则）③ Opus 承重维 A/B 精读得真分差。三件做完才有干净 M1 裁定。
- 正面：名钉死机制端到端跑通无回归、默认管线零改动（flag 隔离）、gen_title 修复随产线生效（书名《假婚真爱：陆总的归宿》干净）。

**M2 — 批量 + 交付门接线**
- Spine 版跑线性甜区 5 本，薄网进交付门，复评承重维。连续验证后，把 §3.6 薄网设为正式 ship_issue。

## 8. 成本 / 风险 / 预期

- **成本**：建 Spine 增量主要在阶段 1 扩抽取（flash，便宜）+ 阶段 2 裁决（少量 pro-verify）。估单本 +¥1-3，相对 ¥21/本（best-of-3）或拒收浪费可忽略。
- **风险**：① 归并裁错（错并毁人物）——靠复用已验证的 verify-gate + reconcile_log 可审计缓解；② 起草不服从硬约束——靠薄网兜 + DRAFT 优先级铁律；③ Spine 抽取漏长尾实体——M0 测覆盖率。
- **预期（严尺）**：承重 40-58 → 65-72（事前消除大宗事实矛盾）；交付均值从 ~64 抬向 70-73；P(≥75) 从 ~10% 升向 ~35-45%。**与 tier3「只 A」情景同量级，但路径更治本**（tier3 的 A2 对账是事后、18% 召回；Spine 是事前、registry 空间裁决）。75-85 仍需选择性交付 + 严选源；稳定 85+ 仍是微调墙。
