# 事件·副线·状态账（A）· 承重事前架构 §M2 续篇

> 版本 v0.1（设计 + M0 计划）｜ 2026-06-17
> 定位：把 Fact Spine 的「事前钉死」从**实体**扩到**事件/副线/状态**，治住实体表覆盖不到的承重残类。
> 前序：`docs/design/fact_spine.md`（实体表已治 名/数值/身份/地点/体系/里程碑）｜`docs/design/load_bearing_measurement.md`（B：先测准）。
> 触发：editor-eval-1 实测——承重差评 100% 落在事件双版本/时序/副线，实体类已被 Spine 治住。

---

## 0. 一句话

实体一致性靠「事前冻结实体登记表 + 起草受硬约束」解了；**事件/因果/副线一致性用同一招**——在 plan 层冻结**事件账（单次、因果序）+ 副线账（设定→回收）+ 状态机（对象状态单调推进）**，让「时间倒退/事件双版本/副线悬空」在结构上不产生，而非事后检测打地鼠。

---

## 1. 问题（editor-eval-1 实测锁定）

实体类已治，承重残类全在事件-因果-副线层：

| 残类 | 编辑实见 | 现状 |
|---|---|---|
| **事件双互斥版本** | 爹「出车祸凑钱」↔「被绑架」（极品全能小村医，承重30） | ❌ 无账 |
| **时序倒退/场景跳跃** | 章缝列表满屏「时间倒退/前章…后章…」（全部 5 本） | △ 章缝**检出**但「缝平≠接对」修不净、不进门 |
| **因果断裂** | 「没说清前因后果」（冰川探墓，承重40） | ❌ 无账 |
| **副线悬空/无铺垫** | 女配无铺垫（冰川探墓）、女角色无介绍（灵气复苏） | ❌ 无账 |

**根因链**（同 fact_spine §1.1）：并发分章 + plan brief 抽象 → 起草自由现编事件顺序/副线 → 跨章各编各的。章缝是**事后缝合**（缝平断面但接错语义），治不了「事件本就该唯一/有序」。

---

## 2. 原理：事件也搬到 registry 空间、前移到 plan

- **prose 空间**：事件顺序/因果/副线散在 21 万字，矛盾不可枚举、检测召回低、缝合引漂。不可解。
- **registry 空间**：`~60-120 行事件账 + 副线账 + 状态机`，矛盾**可枚举**（每事件一次、每副线设定→回收闭环、每对象状态单调），可确定性裁 + 定向 verify，改账不碰正文。**可解。**

并发起草下唯一可行路 = 共享冻结事件骨架 + plan 引用事件 ID + 起草受约束（与 Fact Spine 同构）。

---

## 3. 三本账（落 `schemas.py` 扩展 + plan 接线）

```python
@dataclass
class Event:
    event_id: str            # 唯一,plan/draft 只能引用,禁新造同义事件
    summary: str             # "父亲车祸入院"
    ch: int                  # 唯一发生章(治双版本:车祸 vs 绑架是两个不同 event_id,不可互换)
    causal_after: list[str]  # 前置事件 ID(治因果断裂:绑架需在"已脱离车祸线"之后或互斥)
    unique: bool = True      # 单次发生(治重演/双版本)

@dataclass
class Thread:                # 副线/伏笔/角色线
    thread_id: str
    setup_ch: int            # 引入/铺垫章(治"女配无铺垫":出场前必须有 setup)
    payoff_ch: int | None    # 回收章(治伏笔悬空:None=未回收→交付门拦)
    entity: str              # 关联实体(女配名)

@dataclass
class StateTimeline:         # 关键对象状态机
    entity: str              # "父亲"
    transitions: list[dict]  # [{ch, state}] 单调:健康→车祸→住院... 禁回跳/互斥(治状态矛盾)
```

落 `fact_spine.json` 扩段 `events/threads/state_timelines`，与实体表并列冻结。

---

## 4. 五阶段接线（复用 Fact Spine 管道）

1. **抽取**（扩 `mining` / `macro`）：从源 + macro 节拍抽事件清单 + 副线清单 + 关键对象状态序。
2. **裁决冻结**（registry 空间，一次）：事件去重并规范化（车祸/绑架 → 两个 event_id，定因果序）；副线配 setup→payoff；状态序单调化。
3. **plan 从账生成**：`_plan_macro / _plan_one_chapter` 引用 event_id + 遵守因果序 + 副线在 payoff 前必有 setup → **plan 级就排不出双版本/时间倒退/悬空**（今天靠铁律压不住，从账生成则结构上不产生）。
4. **起草硬约束**（扩 `_control_plane` / `ledger.format_context`）：每章注入「本章事件账（只演这些 event_id、不得新造/重演别章事件）+ 副线状态 + 对象当前状态」。
5. **事后薄网进门**（补 B 的漏洞）：承重分（章缝严重度 + 事件账对账：检出"未在账内的事件/状态矛盾/未回收副线"）→ 进 `ship_issues`，**补上交付门对该类的失明**。

## 6. 与现有件关系 / 边界

| 现有件 | 关系 |
|---|---|
| `fact_spine.json` / `_spine_*` | **扩段**：+events/threads/state_timelines |
| `macro.json` 节拍（已有 beat/act/refs） | **复用**：macro 已是事件雏形，抽事件账的输入 |
| 章缝/控制面重演检测 | **复用+翻转**：从「事后缝合」翻为「对照冻结事件账」的承重分（进门，B §4） |
| `ledger.format_context` | **复用+扩**：注入事件/副线/状态切片 |

**诚实边界**：①事件比实体「软」（边界模糊），抽取/去重难度高于名/数值——M0 先验证可枚举性；②plan 从账生成可排掉结构性双版本，但起草仍可能不听话→薄网兜；③本架构治承重，不抬笔力/人（生成侧天花板）。
