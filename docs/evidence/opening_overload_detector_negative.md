# 开篇信息过载检测器 · 负结果(2026-06-23)

> 任务 2 拟建"开篇信息过载"检测器,补 editor-eval-2 机器盲点(暴君/团宠"开头看不懂、信息量太大",代入感审计漏看)。
> **一发式 LLM-judge 两次迭代方向相反均失败 → 本路不通,搁置待真 R&D。** 代码已回退,src 保持干净。

## 试了什么

`opening_overload_audit(cli, ch1)` + prompt `OPENING_OVERLOAD`,输出 `{过载, 过载分0-100, issues}`,设计为 advisory + 信号特征(signals.opening_overload 占位)。廉价真值探针(¥0.05/轮)对 3-5 本已知人评本验证。

## 两次迭代,反向翻车

| 本(人评) | v1 锚"信息密度" | v2 锚"困惑/跟不上" |
|---|---|---|
| 暴君(承重50,"开头看不懂")应warn | warn 65 ✓ | **ok 10 ✗ 漏** |
| 穿成萌娃(承重70,总76.5最佳)应ok | **warn 75 ✗✗ 最佳本误报最重** | ok 10 ✓ |
| 团宠("第1章复杂")应warn | ok 10 ✗ | ok 10 ✗ |

- **v1 数密度** → 把"信息密"当"信息过载",误杀客观信息密但人类读着顺的最佳本(穿成萌娃 ch1 一次性砸很多名字,但承重70)。
- **v2 锚困惑** → 矫枉过正,全 ok,连真该 warn 的暴君也漏。

## 结论

- **两次方向相反失败 = 不是 prompt 调参问题,是"人类'看不懂'这种细微感知,一发式 LLM-judge 抽不干净"**(systematic-debugging:2次反向失败应质疑方法,非继续hack)。
- **第 2 次坐实 eval-5 铁律**:craft 类感知判断 LLM 不可靠裁决。"密但驾驭住" ≠ "看不懂",LLM 分不清。
- **幸亏全程 advisory 没进门**——否则刚误杀最佳本穿成萌娃。
- **更大的启示(喂测量策略)**:相0 标定已显示可靠信号是**结构化**的(章缝检出 r=-0.50 最强),LLM-judge craft 特征(代入感仅低端可靠、过载不可靠)是弱/噪声特征。质量代理应押结构信号 + 人评飞轮,不押 flaky LLM 检测器。

## 若将来重启(非 naive 重试)

- few-shot 锚定(给 LLM 标注的懵/顺范例)——但易过拟合小样本,10k 泛化存疑。
- **结构化代理**:如"主角目标点出前登场的命名实体数""场景跳切次数"——半确定、可信、可测,优于纯 LLM vibe。
- 先攒更多带人评的"看不懂"标注(重叠评),再判这个维度值不值得、怎么测。
