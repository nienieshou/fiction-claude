# 判别器验证实验(2026-06-12)· test-time compute 路线生死裁定

**实验**:团宠 4 场景(开篇/2普通/结局)各生成 N=6 候选,两个判别器排序,Fable 盲评 Top1 vs Bottom1。¥0.49。

## 结果

| 场景 | 区分度(分跨) | GOLD_PK Top1 = Fable更好? | PICK = GOLD_Top1? |
|---|---|---|---|
| 1 拜师 | 15 | ✓ | ✗ |
| 2 魔族擒师兄 | 8(最低) | ✗(GOLD选反) | ✗ |
| 3 蛇藏袖 | 16 | ✓ | ✗ |
| 4 围攻反派 | 12 | ✓ | ✓ |

- **对照判别器 GOLD_PK 命中率 = 3/4 = 75%**(候选 vs 金标打分)
- **绝对锦标赛 PICK 与 GOLD 一致率 = 1/4 = 25%**

## 三条硬结论

**① 对照判别器(GOLD_PK)有区分力,勉强可用**——75% > 随机 50%,分数跨度 8-16(不是都打同分)。但 75% 不是强信号(4 样本,置信区间宽 ~50-90%),且**唯一失败的场景2 区分度最低(8分)**——候选质量接近时判别器失灵(大 N 的 Top1 vs Top2 可能分不出,边际收益递减)。

**② 重大发现:产线选优用错了判别器**。`_process_scene` 当前用 **PICK**(绝对锦标赛)选 winner,而 PICK 与对照判别器仅 25% 一致、且对照才是对的那个——**PICK ≈ 噪声(Goodhart 实锤,印证设计文档 A5)**。产线现在的"选优"基本是随机选,根本没吃到模型尾部。

**③ test-time compute 路线"半通"**:不是"判别器无效只能微调",也不是"判别器强可大干快上",而是——
- 对照判别器能选出更好的(75%),**大 N 选优有正期望**
- 但增量有限(非 95% 精度),期望是"小幅提分"非"突破到 75"
- **最高 ROI 的零成本改进:产线选优 PICK → GOLD_PK**(让选优真正生效)

## 落地建议(排除微调的可行手段,按 ROI)

### R17 候选(零/低成本,实验直接支持)
1. **产线选优换判别器**(零 API 成本,改 `_process_scene`):PICK → GOLD_PK 打分选最高。这是让"选不靠写"真正生效的前提——当前 PICK 噪声等于没选。**单这一项可能值 +1-2 分**(把随机选优变成 75% 有效选优)。
2. **扩大判别器验证到 20 场景**(¥2):确认 75% 是否稳定。<65% → 路线偏弱;>80% → 值得开全场景大 N。

### R18 候选(中成本,验证通过后)
3. **全场景 N=12 + GOLD_PK 选优**(单本 ¥30-40,¥50 内):吃模型尾部。但区分度低的场景(候选接近)收益递减,实际增量预估 +2-4(把笔力/人从单次 67-68 拉到选优后 70-72)。
4. **few-shot 金标示范注入 DRAFT**(零成本,正交):in-context 拉笔力上限,与大 N 叠加。

### 诚实边界
即使 1+3+4 全做,期望把 A 源从 63 推到 ~67-70,**仍可能差 75 线最后一截**。test-time compute 是"吃单次生成的尾部",尾部最好的那版也受限于 DeepSeek 的能力分布——要稳过 75,大概率仍需微调(路线 K)突破分布本身。但 **test-time compute 是微调前应先榨干的、零运维的杠杆**,且暴露了"产线选优失效"这个真 bug。

## 对终轮结论的修正

`iteration-final-summary.md` 说"笔力/人是生成侧天花板,工程碰不到"——**需修正**:笔力/人的天花板部分是**产线选优失效(PICK噪声)人为压低的**,不全是模型硬上限。换对照判别器 + 大 N 是工程能碰这两维的途径,只是天花板比微调低。

**建议**:转运营态前,先做 R17(换判别器,零成本)+ 扩大验证——这是微调外最后一个未榨干的工程杠杆,且改的是一个实锤 bug(产线用噪声判别器选优)。
