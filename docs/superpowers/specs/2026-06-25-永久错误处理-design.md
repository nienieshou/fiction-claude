# 永久性 API 错误的端到端处理 设计

日期：2026-06-25
状态：已批准设计，待 writing-plans
分支：`fix/永久错误处理`

## 背景

实盘观测：DeepSeek 余额耗尽时，3 本任务在 API 调用处崩溃，崩因相同：
`openai.APIStatusError: Error code: 402 - 'Insufficient Balance'`。暴露两个缺陷：

1. **402 等永久性错误被当成可重试**：`client.complete` 的 `except (APIError, APIConnectionError, APITimeoutError)`（client.py:74）吞下 `APIStatusError`(402/401/403 均其子类)，退避重试 6 次才放弃——余额/认证问题等多久都不会自愈，纯浪费。
2. **崩溃任务被误判为可续跑**：状态逻辑（adapters.py:87-95）无视 `_crash.txt`，崩溃任务（有 _crash.txt、无 report.json）显示 `stalled`/`idle` = "中断·可续跑"。启动自动续跑会重拉 → 再撞同样的永久错误 → 再崩，形成烧时间的循环。

## 决策

### Fix A — 永久性错误判致命，0 重试（`src/hiki/client.py`）

按 `APIStatusError.status_code` 分流：
- **致命集 = {401, 402, 403}**（认证/余额/权限，永不自愈）→ 立即抛 `RuntimeError`，**0 重试**，错误信息含状态码 + 充值/检查提示
- **其余状态码**（5xx 等）+ 连接/超时/空响应 → **照旧重试**（退避+jitter，次数/上限不变）
- **429 RateLimitError** → 保持原有独立处理（已正确退避重试）

异常捕获顺序（openai 类层级：`RateLimitError` < `APIStatusError` < `APIError`；`APIConnectionError`/`APITimeoutError` < `APIError` 但**非** `APIStatusError`）：
1. `except RateLimitError`（429，重试）— 须最先
2. `except APIStatusError as e`：`e.status_code in (401,402,403)` → 致命 raise；否则照旧重试
3. `except (APIError, APIConnectionError, APITimeoutError)`（无响应的连接/超时）→ 照旧重试

### Fix B — 崩溃任务显示 failed，不被自动续跑误 resurrect（`web/backend/adapters.py` + `app.py`）

状态判定加一层（**report.json 仍最优先**，治"崩后续跑成功、_crash.txt 残留共存"——成功覆盖崩溃）：
1. 有 report.json → certified/rejected
2. 活跃任务 → running
3. **有 `_crash.txt`（且无 report）→ `failed`** ← 新增（在 stalled 之前）
4. 有 produce 产物 → stalled
5. 否则 → idle

配套：
- **自动续跑只认 stalled**（app.py:40 不变）→ `failed` 不被自动重拉烧钱
- **手动续跑放行 `failed`**（app.py:142 准入集加 `failed`）→ 充值/修因后可手点重试

## 测试（零真实 API）

`tests/test_client_retry.py`（新）：`monkeypatch.setenv("DEEPSEEK_API_KEY","test")` 构造 Client，替换 `c._c.chat.completions.create` 为计数的 fake，patch `asyncio.sleep` 免真等：
- **致命**：create 抛 `APIStatusError`(402) → `complete()` 抛 `RuntimeError` 且 **create 仅被调 1 次**（无重试）
- **非致命**：create 抛 `APIStatusError`(500) → 重试耗尽后抛，**create 被调 6 次**
- （可选）429 `RateLimitError` 仍重试，不立即致命

`tests/test_web_normalize.py` 或 `test_web_adapters.py`（追加）：
- 造 out_dir：有 `_crash.txt` + produce 产物 + **无 report.json** → `dir_to_book` 状态 = `failed`
- 造 out_dir：有 `_crash.txt` + **有 report.json**（崩后续跑成功）→ 状态仍 = certified/rejected（report 优先）

`tests/test_web_*`：resume 端点对 `failed` 放行（status 准入含 failed）。

## 非目标（YAGNI）

- 不改瞬时错误的重试次数/退避曲线
- 不自动删 `_crash.txt`（保留供诊断；report.json 优先已治共存）
- 不改 best-of / 自动续跑的 stalled 语义
- 不加心跳/超时熔断（缺陷分析里的 gap #2，另案）
